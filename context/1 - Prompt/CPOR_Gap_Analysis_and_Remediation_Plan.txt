SECTION 1 — PROJECT BACKGROUND
================================================================================

The CPOR Global Industry Catalog is a 3-tier Dataverse extension layered on
top of the existing CPOR Sales solution:

  Tier 1 — TAXONOMY     cpor_IndustryCode, cpor_ComplianceDomain, Territory (OOB extended)
  Tier 2 — CROSSWALK    cpor_IndustryDomainMap, cpor_TerritorySourceMap, cpor_RegulatorySource
  Tier 3 — RESOLUTION   cpor_RegulatoryRegistration

The resolution tier is the operational heart of the system. Industry Team
analysts create and maintain cpor_RegulatoryRegistration records, which are
consumed by the CPOR AI pipeline at Stage 0 to ground regulatory hypotheses
for sellers.

Key artifacts:
  - Model Driven App: CPOR Industry Catalog
  - BPF:             CPOR — Registration Qualification (5 stages)
  - Power Automate:  4 Cloud Flows (Flows 1–4)
  - Web Resources:   4 HTML pages + 7 JS modules
  - Schema:          7 JSON schema files (see /schema/tables/)
  - Design guides:   8 GII guides + 8 CRM guides (see /context/2 - Design/)

The 4-step wizard (cpor_new_registration_dialog.js) is the primary intake
mechanism for analysts creating new cpor_RegulatoryRegistration records.

SECTION 2 — GAP ANALYSIS
================================================================================
10 gaps were identified across the design documentation, schema, and code.
Gaps are rated CRITICAL, HIGH, or MEDIUM based on operational impact.

--------------------------------------------------------------------------------
GAP 1 — No Research Initiation Context  [MEDIUM]
--------------------------------------------------------------------------------
Description:
  The cpor_RegulatoryRegistration entity has no fields to capture how a
  research need was identified, what market context motivated it, or who
  requested it. This makes audit trails incomplete and prevents seller-demand
  analytics.

Impact:
  - No way to trace whether a registration was created proactively (regulatory
    monitoring) or reactively (seller request).
  - No owner accountability for intake beyond the Dataverse OwnerId.
  - Cannot measure which sellers are driving catalog expansion.

Missing fields:
  cpor_intakesource   (Choice — how the need was identified)
  cpor_marketcontext  (Text — brief market context description)
  cpor_requestedby    (Lookup → systemuser — who requested the research)

Resolution: Phase E — add 8 new Intake and Sales Alignment fields.

--------------------------------------------------------------------------------
GAP 2 — Wizard Bypasses BPF Qualification  [CRITICAL]
--------------------------------------------------------------------------------
Description:
  The 4-step wizard creates records via direct OData POST. Step 3 previously
  exposed a status dropdown (cpor_registrationstatus) that allowed analysts
  to select "Active" at intake, bypassing all 5 BPF stages and skipping the
  Territory Manager approval gate.

Impact:
  - Records could enter the pipeline as Active without TM approval.
  - Pipeline hypothesis grounding could use unverified registrations.
  - BPF data integrity is meaningless if status can be set freely at create.

Root cause:
  cpor_new_registration_dialog.js step3HTML() rendered STATUS_OPTS as a
  <select> element. buildPayload() passed the user-selected value directly.

Resolution: Phase A
  - Removed status dropdown from step3HTML().
  - Added informational notice banner explaining Pending-only intake.
  - Hardcoded cpor_registrationstatus = 154080002 (Pending) in buildPayload().

--------------------------------------------------------------------------------
GAP 3 — TM Handoff Fragile  [MEDIUM]
--------------------------------------------------------------------------------
Description:
  When a registration reaches BPF Stage 4 (Pending TM Review), Flow 4 routes
  an approval to territory.managerid. Three weaknesses exist:
  (a) No notification to the submitting analyst on approval or rejection.
  (b) No SLA tracking — no field records when Stage 4 was submitted.
  (c) No escalation if the TM does not respond (approval hangs indefinitely).

Impact:
  - Analyst has no visibility after submitting for TM review.
  - TM inaction blocks pipeline grounding silently.
  - No data to report on approval latency or SLA compliance.

Resolution: Phase E — Flow 4 enhancements:
  - Added cpor_stage4submitteddate stamp (Step 1b in Flow 4).
  - Added 7-day parallel timeout branch with escalation notification.
  - Added analyst notification on both approval and rejection outcomes.

--------------------------------------------------------------------------------
GAP 4 — No Duplicate Check in Wizard  [HIGH]
--------------------------------------------------------------------------------
Description:
  The wizard performs no pre-flight duplicate check before POSTing to
  Dataverse. HTTP 412 errors from Dataverse's duplicate detection rules are
  caught only generically (any error → "An unexpected error occurred."), giving
  analysts no actionable guidance.

Impact:
  - Duplicate registrations can proliferate if the analyst does not manually
    search before creating.
  - The error message on HTTP 412 gives no context about which existing records
    conflict.

Resolution: Phase A
  - Added checkDuplicate() — async OData query filtering on
    _cpor_industrycode_value + _cpor_territory_value + _cpor_compliancedomain_value
    (excluding Superseded records).
  - Added showDuplicateWarning() — modal overlay listing conflicting records
    with Cancel / Create Anyway options.
  - Refactored handleSave() to orchestrate duplicate check → warning → doCreate().
  - Added specific 412/DuplicateRecord error handling in doCreate() with a
    clear, actionable message.

--------------------------------------------------------------------------------
GAP 5 — No Crosswalk Context in Wizard  [HIGH]
--------------------------------------------------------------------------------
Description:
  Step 1 of the wizard collects Industry Code + Territory + Compliance Domain,
  but provides no feedback showing whether the selected Compliance Domain is
  mapped to the selected Industry Code in cpor_IndustryDomainMap. Analysts may
  create registrations for domain-industry combinations that have no crosswalk
  entry, weakening the catalog's structure.

Impact:
  - Silent omission: no warning when a domain is not in the industry crosswalk.
  - Analysts who are unfamiliar with the crosswalk structure may create
    orphaned registrations.

Resolution: Phase B
  - Added loadCrosswalkContext() — fetches cpor_IndustryDomainMap rows for the
    selected industry code (cached per industry).
  - Added renderCrosswalkCtx() — displays a context panel below the taxonomy
    selects showing:
      - Green tick if the selected domain is in the crosswalk (with priority).
      - Amber warning if the domain is NOT in the crosswalk.
      - Amber warning if no domains are mapped to the industry at all.
      - List of all mapped domains for the industry, with priority badges.
  - Added hideCrosswalkCtx() — clears the panel when fields are incomplete.
  - Updated step1HTML() to inject the #cpor-wiz-ctx panel.
  - Updated onStep1Input() to reset the cache on industry change and call
    loadCrosswalkContext() on every Step 1 input event.

--------------------------------------------------------------------------------
GAP 6 — Multi-Source Evidence Not Captured  [MEDIUM]
--------------------------------------------------------------------------------
Description:
  cpor_regulatorysource is a single optional Lookup to cpor_RegulatorySource.
  There is no mechanism to record supplementary reference URLs (e.g. multiple
  regulatory body pages, enforcement notices, guidance documents).

Impact:
  - Analysts must choose a single "primary" source and lose other references.
  - Verification quality is harder to assess without multiple citations.

Resolution: Phase E
  - Added cpor_additionalsourceurls (TextArea 2000) to the entity schema guide
    and to CRM Guide 03 Section 3 (after Legislation URL).
  - Added to wizard buildPayload().

--------------------------------------------------------------------------------
GAP 7 — No Seller Feedback Loop  [LOW — DEFERRED]
--------------------------------------------------------------------------------
Description:
  Sellers consuming pipeline output cannot flag a regulatory hypothesis as
  incorrect, outdated, or irrelevant to their account context. There is no
  in-product mechanism to surface seller feedback back to Industry Team
  analysts for catalog correction.

Impact:
  - Catalog quality degrades silently over time if sellers cannot signal errors.
  - Industry analysts have no data on which registrations are generating
    inaccurate hypotheses.

Resolution: Phase F (deferred)
  - Requires cpor_Hypothesis entity changes and pipeline team coordination.
  - Out of scope for this remediation cycle.

--------------------------------------------------------------------------------
GAP 8 — No Sales Motion Relevance Fields  [MEDIUM]
--------------------------------------------------------------------------------
Description:
  cpor_RegulatoryRegistration has no fields linking a regulation to Microsoft
  solution areas, buyer personas, or seller-ready narrative. Analysts cannot
  annotate registrations with the sales context that would help sellers use the
  catalog most effectively.

Impact:
  - Sellers receive a regulation name and applicability rule but no Microsoft
    framing.
  - Sales leaders cannot segment the catalog by solution area or buyer persona.

Resolution: Phase E
  - Added cpor_solutionarea (Text 100), cpor_buyerpersona (Text 100),
    cpor_salesnarrative (TextArea 500) to the entity and wizard.
  - Added Section 7 "Sales Alignment" to the CRM Guide 03 form layout.

GAP 9 — Field Name Mismatches in Design Guides  [HIGH]
--------------------------------------------------------------------------------
Description:
  Multiple implementation guides reference field names that do not match the
  deployed schema. The deployed names are the correct ones (confirmed from
  schema JSON files); the guide names are earlier working names that were
  changed before or during deployment but the guides were not updated.

  GUIDE NAME (incorrect)   →   DEPLOYED NAME (correct)
  ─────────────────────────────────────────────────────
  cpor_analystnotes        →   cpor_regulatorynotes
  cpor_approvedby          →   cpor_verifiedby
  cpor_mandatory           →   cpor_ismandatory
  cpor_status              →   cpor_registrationstatus
  cpor_industrycodeid      →   cpor_industrycode  (FK logical name)
  cpor_territoryid         →   cpor_territory     (FK logical name)
  cpor_compliancedomainid  →   cpor_compliancedomain (FK logical name)
  cpor_regulatorysourceid  →   cpor_regulatorysource (FK logical name)

  Additionally, the Stage 0 OData query in GII Guide 04 used wrong attribute
  names:
    _cpor_industrycodeid_value  →  _cpor_industrycode_value
    _cpor_territoryid_value     →  _cpor_territory_value
    cpor_compliancedomainid($select=...) in $expand → cpor_ComplianceDomain(...)
    cpor_territoryid($select=...)        in $expand → cpor_Territory(...)
    cpor_territoryid/cpor_jurisdictionlevel in $orderby → cpor_Territory/cpor_jurisdictionlevel

  Business Rules in CRM Guide 05 referenced wrong field names:
    BR1: disable list used cpor_industrycodeid, cpor_mandatory, etc.
    BR1: FIELD THAT REMAINS EDITABLE referenced cpor_analystnotes
    BR2: condition used cpor_mandatory instead of cpor_ismandatory
    Both BRs: cpor_status used instead of cpor_registrationstatus

Impact:
  - Business Rules configured from these guides will reference non-existent
    fields and silently fail to lock/require the intended fields.
  - Data Import steps using wrong field names will fail.
  - Any developer or admin following the guides will create incorrect
    configurations.

Resolution: Phase C
  - GII Guide 04: all field name corrections applied.
  - CRM Guide 03: header field, Section 1 FK names, Section 4 field names,
    Section 6 stale deployment note removed.
  - CRM Guide 05: BR1 condition, disable list, editable field, verification
    steps; BR2 condition and portal steps.
  - CRM Guide 07: Added FIELD NAME REFERENCE TABLE to PREREQUISITE section.

GAP 10 — Freshness Gate Inconsistency  [MEDIUM]
--------------------------------------------------------------------------------
Description:
  GII Guide 00 Decision 7 stated "90 days (configurable)" as the catalog
  freshness threshold. All code (cpor_xrm_client.js isStale function), all
  flows, and all CRM guides consistently use 180 days. The GII Guide 00 figure
  was a stale draft value.

  Additionally, GII Guide 00 listed "(6)" new optionsets, but 8 had actually
  been defined (cpor_regulatortype #7 and cpor_registrationstatus #8 added
  after the overview was last updated). After Phase E, the count is 9.

Impact:
  - Administrators configuring Business Rules from the guide would set the
    wrong freshness threshold.
  - The optionset count discrepancy causes confusion when deploying.

Resolution: Phase D
  - GII Guide 00: Decision 7 corrected to 180 days with env var reference.
  - GII Guide 00: Optionset count corrected to 9; added cpor_regulatortype and
    cpor_registrationstatus entries with value counts and descriptions.
  - CRM Guide 00: Cross-reference updated to cite Decision 7 and
    CATALOG_FRESHNESS_DAYS environment variable.
