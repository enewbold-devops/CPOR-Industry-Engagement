Implementation Guide 04: Resolution Entity (cpor_RegulatoryRegistration)
Date: 2026-06-13
================================================================================

PURPOSE
-------
Create cpor_RegulatoryRegistration — the core "grounding record" for the
catalog tier. Each record represents one specific regulation (or regulatory
obligation) applicable to a given industry in a given territory/jurisdiction
under a given compliance domain.

Stage 0 queries this entity to build the candidateRegulations[] array that
grounds Stage 4 regulatory verification and populates FK fields on hypotheses.

PREREQUISITES
-------------
Phase 1 (optionsets), Phase 2 (taxonomy entities), and Phase 3 (crosswalk
entities) must be deployed first. This entity has FKs to all 3 taxonomy
entities plus the OOB territory entity.


================================================================================
ENTITY — cpor_RegulatoryRegistration
================================================================================
Display Name:       Regulatory Registration
Plural Name:        Regulatory Registrations
Entity Set Name:    cpor_regulatoryregistrations
Schema Name:        cpor_RegulatoryRegistration
Primary Field:      cpor_name
Description:        A curated record of a specific regulatory obligation,
                    legislation, or compliance requirement applicable to a
                    particular industry code in a particular territory/
                    jurisdiction under a particular compliance domain.
                    These records ground AI-generated regulatory claims and
                    provide authoritative FK references for hypotheses.

OWNERSHIP:          User or Team
AUDIT:              Enabled
CHANGE TRACKING:    Enabled (required for Stage 0 date-filtered query)

================================================================================
DUPLICATE DETECTION RULE
================================================================================
Rule Name: "Duplicate Regulatory Registration"
Fields:    cpor_industrycodeid AND cpor_territoryid AND cpor_legislationname
Match:     Exact on all 3 fields (case-insensitive on cpor_legislationname)
Action:    Warn on create and import

This prevents the same law from being entered twice for the same
industry × territory combination.

================================================================================
STAGE 0 QUERY PATTERN
================================================================================
referenceResolver.js builds candidateRegulations[] using:

  const freshnessDate = new Date();
  freshnessDate.setDate(freshnessDate.getDate() - CATALOG_FRESHNESS_DAYS);
  const freshnessCutoff = freshnessDate.toISOString().split('T')[0]; // YYYY-MM-DD

  // Build territory filter across the jurisdiction chain
  const territoryFilter = jurisdictionChain
    .map(t => `_cpor_territoryid_value eq ${t.territoryid}`)
    .join(' or ');

  GET /cpor_regulatoryregistrations
    ?$filter=
      _cpor_industrycodeid_value eq '{industryCodeGuid}'
      and ({territoryFilter})
      and cpor_status eq 'Active'
      and cpor_lastverifieddate ge {freshnessCutoff}
    &$expand=
      cpor_compliancedomainid($select=cpor_domaincode,cpor_name),
      cpor_territoryid($select=cpor_territorycode,cpor_jurisdictionlevel)
    &$select=
      cpor_regulatoryregistrationid,
      cpor_name,
      cpor_legislationname,
      cpor_applicabilityrule,
      cpor_mandatory,
      cpor_riskrating,
      cpor_legislationurl,
      cpor_effectivedate,
      cpor_compliancedeadline
    &$orderby=cpor_territoryid/cpor_jurisdictionlevel desc

Returns records sorted most-specific first (Country before Region before Global).
LLM receives these as the `candidateRegulations` array in the grounding context.

Environment variable:
  CATALOG_FRESHNESS_DAYS=180  (set in local.settings.json and Azure Function App settings)
  If a record's cpor_lastverifieddate is older than this, it is excluded.
  If this variable is not set, default to 180.

================================================================================
@odata.bind PATTERN FOR HYPOTHESIS FK
================================================================================
When dataverseWriter.js creates a cpor_Hypothesis record and the LLM has
emitted a regulatoryRegistrationId in the hypothesis JSON, the bind is:

  "cpor_regulatoryregistration@odata.bind":
    "/cpor_regulatoryregistrations(${hypothesis.regulatoryRegistrationId})"

The regulatoryRegistrationId is a GUID returned by Stage 0 in the
candidateRegulations[] array. Stage 4/5/6 pass it through the hypothesis
object if applicable.

Full FK bind block in dataverseWriter.js createHypotheses (see Guide 09):

  ...(h.regulatoryRegistrationId
    ? { "cpor_regulatoryregistration@odata.bind":
          `/cpor_regulatoryregistrations(${h.regulatoryRegistrationId})` }
    : {}),
  ...(h.complianceDomainCode
    ? { "cpor_compliancedomain@odata.bind":
          `/cpor_compliancedomains(cpor_domaincode='${h.complianceDomainCode}')` }
    : {}),
  ...(h.territoryCode
    ? { "cpor_territory@odata.bind":
          `/territories(cpor_territorycode='${h.territoryCode}')` }
    : {}),
