================================================================================
CPOR DURABLE — INDUSTRY CRM IMPLEMENTATION
Implementation Guide 03: Forms
Date: 2026-06-15
================================================================================

PURPOSE
-------
Define the main form layouts for all catalog entities used by the Global
Industry Team. Forms control what analysts see and edit when they open a
record. Well-structured forms reduce error, enforce workflow discipline,
and surface compliance context clearly.

This guide specifies:
  - cpor_RegulatoryRegistration Main Form (primary — most detailed)
  - cpor_IndustryDomainMap Main Form
  - cpor_TerritorySourceMap Main Form
  - territory Main Form extension (CPOR fields section added to OOB form)
  - cpor_RegulatorySource, cpor_IndustryCode, cpor_ComplianceDomain
    (minor form additions — brief field configuration notes)

PREREQUISITE: Guide 01 (App Module) complete. GII Guides 00-07 deployed.

PORTAL STEPS (applies to all forms)
-------------------------------------
1. In make.powerapps.com, open the CPOR solution.
2. Select Objects > Tables, then select the target table.
3. Select Forms from the table's components list.
4. Open the existing "Information" Main form (every table has one by default).
5. Use the form editor to add sections, tabs, and fields as specified.
6. Select Save, then Publish.

NOTE: Always edit the existing Main form rather than creating a new one,
unless the existing form is locked (managed solution). Creating a duplicate
form requires configuring it as the default in the app, which adds steps.


================================================================================
FORM 1: cpor_RegulatoryRegistration — Main Form
================================================================================

FORM NAME: Information (existing default — edit in place)
HEADER FIELDS (always visible, top of form):
  cpor_name                  (primary field — legislation short name)
  cpor_registrationstatus    (Registration Status choice field — quick-scan pipeline eligibility)

TAB STRUCTURE: Single tab (no tab bar needed — all sections in one scroll)

SECTION 1: Taxonomy Anchors
---------------------------
Purpose: The "who does this apply to" trinity. Required FKs that anchor
         every regulation to a specific industry, territory, and domain.
         Analysts must set all three before the record is useful to the pipeline.

Layout: 2-column

  Left column                       Right column
  --------------------------------  --------------------------------
  Industry Code *                   Territory *
  (cpor_industrycode)               (cpor_territory)
  Compliance Domain *               Regulatory Source
  (cpor_compliancedomain)           (cpor_regulatorysource)
  Requested By
  (cpor_requestedby)

Field labels use the display names from GII Guide 04.
Fields marked * are required (RequiredLevel = Required per GII Guide 04).

SECTION NOTE: Add a section description (visible to analyst in form):
  "Required: Set all three taxonomy anchors before saving. Industry Code and
  Territory are read by the pipeline at Stage 0. Compliance Domain drives
  hypothesis classification at Stage 4."


SECTION 2: Regulator Identity
------------------------------
Purpose: Identifies who issues or oversees this regulation. Provides
         context for executive conversations and compliance research.

Layout: 2-column

  Left column                       Right column
  --------------------------------  --------------------------------
  Regulator / Body Name             Regulator Type
  (cpor_regulatorbodyname)          (cpor_regulatortype)
  Regulator Website URL             Intake Source
  (cpor_regulatorwebsiteurl)        (cpor_intakesource)

Layout: 1-column (below the 2-column rows)
  Market Context (cpor_marketcontext)    [full width, single-line text]


SECTION 3: Legislation Details
--------------------------------
Purpose: The specific law or standard. Most important for analyst curation
         quality — this is what appears in seller-facing hypothesis grounding.

Layout: 2-column header + full-width fields below

  Row 1 (2-column):
  Left: Legal Type (cpor_legaltype)     Right: (empty — visual balance)

  Row 2 (full-width, 1-column):
  Legislation Name (cpor_legislationname)       [full width]
  Legislation URL (cpor_legislationurl)         [full width]
  Additional Sources (cpor_additionalsourceurls) [full width, multiline, 3 rows]
  Applicability Rule (cpor_applicabilityrule)   [full width, multiline, 4 rows]

SECTION NOTE: "Applicability Rule should be plain-language. Example: 'Applies
when the organization processes personal data of EU residents, regardless of
where the organization is located.'"


SECTION 4: Risk and Status
---------------------------
Purpose: Operational status and risk severity. Risk Rating feeds the
         pipeline's regulatoryExposure score in Stage 7.
         Registration Status drives BPF routing and pipeline eligibility.

Layout: 2-column

  Left column                       Right column
  --------------------------------  --------------------------------
  Risk Rating                       Mandatory
  (cpor_riskrating)                 (cpor_ismandatory)
  Registration Status               (empty)
  (cpor_registrationstatus)

FIELD NOTE — cpor_registrationstatus: This is now a Choice field (Optionset 8
from GII Guide 01). Values: Active / Superseded / Pending / Pending TM Review.
Sellers and analysts see a dropdown; no manual text entry required.
Do not display the deprecated cpor_status text field on any form.

NOTE: Business Rule 1 reads cpor_registrationstatus to lock Superseded records.
Ensure cpor_registrationstatus is visible on all forms where Business Rule 1 runs.


SECTION 5: Compliance Dates
-----------------------------
Purpose: Critical for freshness management and deadline alerting.
         cpor_lastverifieddate is the primary operational field —
         it controls whether the pipeline uses this record (180-day gate).

Layout: 2-column, with cpor_lastverifieddate visually emphasized

  Left column                       Right column
  --------------------------------  --------------------------------
  Effective Date                    Compliance Deadline
  (cpor_effectivedate)              (cpor_compliancedeadline)
  Last Verified Date  ← HIGHLIGHT   Next Significant Date
  (cpor_lastverifieddate)           (cpor_nextsignificantdate)

HIGHLIGHT cpor_lastverifieddate:
  In the field properties within the form editor, set the field label font
  to Bold. This draws analyst attention to the most operationally important
  field on the form.
  Add a field description: "Set this date every time you confirm this record
  is still current. Records not verified within 180 days are excluded from
  pipeline grounding."

SECTION NOTE: "Compliance Deadline: Leave blank if the regulation is already
in force with no upcoming deadline. Set only if there is a specific date
by which organizations must comply."


SECTION 6: Analyst Notes
--------------------------
Purpose: Freeform scratchpad for analyst context, change history notes,
         research links not covered by other fields, and flags for follow-up.
         This section remains editable when a record is Superseded (Business
         Rule in Guide 05 locks all other sections for Superseded records).

Layout: 1-column, full width

  Regulatory Notes (cpor_regulatorynotes)

NOTE: cpor_regulatorynotes is the deployed field for analyst/TM notes
(Display Name: "Regulatory Notes", TextArea 4000 chars). This section
remains editable when a record is Superseded (Business Rule 1 locks all
other sections but leaves cpor_regulatorynotes editable).


SECTION 7: Sales Alignment
---------------------------
Purpose: Seller-facing context fields that link the regulation to a Microsoft
         sales motion. These fields are completed post-BPF by the analyst or
         Territory Manager. Not required for pipeline grounding but visible
         to sellers browsing the catalog.

Layout: 2-column header + full-width below

  Row 1 (2-column):
  Left: Solution Area (cpor_solutionarea)   Right: Buyer Persona (cpor_buyerpersona)

  Row 2 (full-width):
  Sales Narrative (cpor_salesnarrative)   [full width, multiline, 3 rows]

SECTION NOTE: "Sales Alignment fields are optional analyst annotations.
  Sellers: Use Sales Narrative as a conversation starting point — verify
  accuracy with your account team before using in customer-facing materials."

FORM FOOTER
-----------
No custom footer fields needed. The OOB audit fields (Created By, Modified By,
Modified On) are visible in the standard form footer.

FORM COMMAND BAR: No custom command bar changes required for this release.

FORM 4: territory (OOB) — CPOR Extensions Section
================================================================================

PURPOSE: Add a dedicated CPOR section to the OOB territory Main form so
         analysts can edit the 5 custom fields without leaving the form.
         The rest of the OOB territory form is left unchanged.

PORTAL STEPS:
1. Navigate to Objects > Tables > Territory.
2. Select Forms > open the "Territory" Main form.
3. Scroll to the bottom of the form (after the existing OOB sections).
4. Add a new Section named "CPOR Catalog Properties".
5. Add the 5 custom fields to this section.

SECTION: CPOR Catalog Properties (2-column)
  Left: Territory Code (cpor_territorycode)
  Right: Territory Type (cpor_territorytype)
  Left: Jurisdiction Level (cpor_jurisdictionlevel)
  Right: ISO Country Code (cpor_isocountrycode)
  Left: Source URL (cpor_sourceurl)
  Right: (empty)

SECTION NOTE: "CPOR fields. Territory Code is the alternate key used in
pipeline grounding (e.g. 'EMEA-EU'). Do not change Territory Code after
initial import — it is used as a stable reference in orchestrator queries."

IMPORTANT: Only add the CPOR Catalog Properties section. Do NOT modify
any existing sections of the OOB territory form. OOB territory fields
(name, parentterritoryid, managerid, etc.) remain in their existing positions.

Publish the territory form after saving.
